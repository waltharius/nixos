* Overview

This document describes the procedure for upgrading this flake-based multi-host NixOS
configuration from 25.11 to 26.05, followed by a retrospective of every problem
encountered during the upgrade of the ~azazel~ laptop on 2026-06-01.

NixOS 25.11 reaches end-of-life on *2026-06-30*. All hosts should be upgraded before
that date.

* Upgrade Procedure

** 1. Edit ~flake.nix~ — bump pinned inputs

Three inputs in this repo are pinned to a specific NixOS release branch.
Change all three from ~25.11~ to ~26.05~:

#+begin_src nix
# Before:
nixpkgs.url     = "github:NixOS/nixpkgs/nixos-25.11";
home-manager.url = "github:nix-community/home-manager/release-25.11";
nixvim.url      = "github:nix-community/nixvim/nixos-25.11";

# After:
nixpkgs.url     = "github:NixOS/nixpkgs/nixos-26.05";
home-manager.url = "github:nix-community/home-manager/release-26.05";
nixvim.url      = "github:nix-community/nixvim/nixos-26.05";
#+end_src

The remaining inputs (~nixpkgs-unstable~, ~nix-flatpak~, ~sops-nix~, ~disko~) track
~HEAD~ and do not need to be changed.

** 2. Update ~flake.lock~

#+begin_src bash
cd ~/nixos
nix flake update
git add flake.nix flake.lock
git commit -m "chore: bump nixpkgs/home-manager/nixvim to 26.05"
#+end_src

** 3. Upgrade laptops (local ~nixos-rebuild~)

Prefer ~boot~ over ~switch~ for major upgrades — if something goes wrong after reboot,
the previous generation is still selectable from the GRUB menu.

#+begin_src bash
# Build only (no activation) — verify there are no eval errors first
sudo nixos-rebuild build --flake .#azazel

# Apply on next reboot (safer for major upgrades)
sudo nixos-rebuild boot --flake .#azazel
sudo reboot

# Or apply immediately (riskier but faster feedback)
sudo nixos-rebuild switch --flake .#azazel
#+end_src

** 4. Upgrade servers (Colmena)

After ~flake.lock~ is updated and committed, deploy via Colmena.
Recommended order: test containers first, physical server last.

#+begin_src bash
# Test container — low risk
colmena apply --on nixos-test

# Production containers
colmena apply --on @lxc

# Physical bare-metal server (altair) — do last, ensure console access first
colmena build --on altair   # build only, verify before applying
colmena apply --on altair
#+end_src

~altair~ has ~buildOnTarget = true~ in ~colmena.nix~, so the build happens on the
server itself. Ensure ~altair~ has enough free space in ~/nix/store~ before upgrading.

** 5. Recommended upgrade order (risk-based)

1. ~nixos-test~ (LXC, no consequences if it breaks)
2. ~azazel~ or ~sukkub~ (laptops — physical access + GRUB rollback)
3. ~cloud-apps~ (LXC production — after testing on ~nixos-test~)
4. ~altair~ (bare-metal — last; requires console/IPMI access for recovery)

* Breaking Changes in 26.05 (affecting this repo)

** NetworkManager / wpa_supplicant integration change

*** What changed

In 25.11, NetworkManager spawned its own internal ~wpa_supplicant~ process.
In 26.05, NetworkManager delegates ~wpa_supplicant~ to the NixOS
~networking.wireless~ module. This means:

- If ~networking.wireless.enable = false~ (or ~lib.mkForce false~) is set anywhere
  in the configuration, ~wpa_supplicant.service~ is never generated by NixOS.
- NetworkManager then fails to start WiFi with:
  : Failed to D-Bus activate wpa_supplicant service

*** Symptom

After upgrading, WiFi is completely dead. No ~wpa_supplicant.service~ unit exists:

#+begin_src bash
systemctl status wpa_supplicant.service
# → Unit wpa_supplicant.service not found.
#+end_src

*** Root cause in this repo

~modules/system/wifi.nix~ contained:

#+begin_src nix
networking.wireless.enable = lib.mkForce false;
#+end_src

This was originally added to prevent a conflict between NetworkManager and the
legacy ~wpa_supplicant~ module. In 26.05 it became the cause of the breakage.

*** Fix

Remove the line entirely. ~networking.networkmanager.enable = true~ is sufficient;
NetworkManager handles the ~wpa_supplicant~ integration automatically.

Relevant NixOS breaking-changes announcement:
https://discourse.nixos.org/t/breaking-changes-announcement-for-unstable/17574/116

*** Workaround (no network, no rebuild possible)

If the system is already upgraded and WiFi is dead with no Ethernet available:

#+begin_src bash
# Try to manually unblock and start wpa_supplicant
sudo systemctl unmask wpa_supplicant
sudo systemctl start wpa_supplicant
sudo systemctl restart NetworkManager

# If that fails, find the binary and run it manually
find /nix/store -name "wpa_supplicant" -type f 2>/dev/null | head -3
sudo wpa_supplicant -B -i wlan0 -c /dev/null -D nl80211
sudo systemctl restart NetworkManager
nmcli device wifi connect "YOUR_SSID" --ask
#+end_src

Once online: ~git pull~ and ~sudo nixos-rebuild switch --flake .#<host>~.

** ~system.nixos.label~ conflict (~mkDefault~ priority collision)

*** What changed

In 26.05, ~nixpkgs/nixos/modules/misc/label.nix~ sets ~system.nixos.label~ using
~lib.mkDefault~ (priority 1000). This repo's ~modules/system/base.nix~ also used
~lib.mkDefault~ for the same option, causing a hard conflict:

: error: The option `system.nixos.label' has conflicting definition values:
:   - In `.../modules/system/base.nix': "manual-<rev>"
:   - In `.../label.nix': "26.05.YYYYMMDD.<rev>"

Two ~lib.mkDefault~ definitions at the same priority cannot be automatically
resolved by the Nix module system — it requires an explicit priority tie-breaker.

*** Fix

Changed ~lib.mkDefault~ to ~lib.mkOverride 900~ in ~base.nix~:

#+begin_src nix
system.nixos.label = lib.mkOverride 900 (
  if self ? rev
  then "manual-${builtins.substring 0 8 self.rev}"
  else "manual-dirty"
);
#+end_src

Priority ladder (lower number = higher priority):
- ~lib.mkForce~   = 50
- ~lib.mkOverride 900~ = 900  ← our definition (wins over nixpkgs default)
- ~lib.mkDefault~ = 1000       ← nixpkgs ~label.nix~ default

* Self-Introduced Bugs During This Upgrade Session

These were mistakes made while applying fixes — documented here so they are not
repeated in future upgrades.

** Incorrect ~sopsFile~ path and format in ~wifi.nix~

*** What happened

While removing ~networking.wireless.enable = lib.mkForce false~ from ~wifi.nix~,
the ~sops.secrets~ block was also accidentally edited, changing:

#+begin_src nix
# Original (correct):
sopsFile = ../../secrets/wifi.env;
format   = "dotenv";

# Accidentally changed to (wrong):
sopsFile = ../../secrets/wifi.yaml;
format   = "yaml";
#+end_src

The file ~secrets/wifi.yaml~ does not exist in the repository (only ~secrets/wifi.env~
exists). This caused the next ~nixos-rebuild~ to fail with:

: error: Path 'secrets/wifi.yaml' does not exist in Git repository

*** Fix

Reverted ~sopsFile~ and ~format~ back to their original values in a follow-up commit.

*** Lesson

When editing a file to apply a targeted fix, always diff the change before
committing (~git diff~) and verify that only the intended lines were modified.

** ~system.nixos.label~ set unconditionally with ~lib.mkForce~

*** What happened

To add a dated label to auto-upgrade generations, ~auto-upgrade.nix~ was given:

#+begin_src nix
system.nixos.label = lib.mkForce (
  let buildDate = builtins.readFile (pkgs.runCommand "build-date" {} ''
    echo -n $(date -d @${toString self.lastModified} +%d-%m-%Y) > $out
  '');
  in "auto-upgrade-${buildDate}"
);
#+end_src

Because this is a Nix option evaluated at *build time* (not at service runtime),
~lib.mkForce~ made *every* rebuild — manual or automated — receive the
~auto-upgrade-DD-MM-YYYY~ label. The intent was to tag only automated upgrades.

*** Symptom

#+begin_src bash
nixos-rebuild list-generations
# Generation 136  auto-upgrade-01-06-2026  ← manual rebuild, wrongly labelled
# Generation 135  auto-upgrade-31-05-2026  ← manual rebuild, wrongly labelled
#+end_src

*** Fix

Removed ~system.nixos.label~ from ~auto-upgrade.nix~ entirely. Instead, the
~nixos-upgrade~ systemd service sets the ~NIXOS_LABEL~ environment variable in its
~preStart~ script. ~nixos-rebuild~ reads this variable at runtime and uses it as the
generation label — so only builds triggered by the ~nixos-upgrade~ service receive
the dated tag.

#+begin_src bash
# In preStart of systemd.services.nixos-upgrade:
export NIXOS_LABEL="auto-upgrade-$(date +%d-%m-%Y)"
#+end_src

Result:
- Manual ~nixos-rebuild switch~: label = ~manual-<8-char git rev>~ (from ~base.nix~)
- Auto-upgrade service run: label = ~auto-upgrade-DD-MM-YYYY~
